# AGENTS.md

## 项目定位

本项目为短剧 AI 生产流程 V2，核心目标是：

- 将一集短剧脚本、人物图、场景图、道具图统一转化为可审核的分镜规划。
- 批量生成多宫格分镜草图提示词与对应视频提示词。
- 保证草图提示词与视频提示词来自同一份 `shot_block_plan.json`，避免两套智能体思路不一致。
- 保留人工审核节点，由用户负责最终判断、测试和平台生成操作。

当前核心流程位于：

```text
17_cherry_pipeline/
```

## 输出语言与沟通风格

- 默认使用简体中文。
- 结论先行，表达简洁、清楚、可执行。
- 涉及代码、文件、流程变更时，先说明准备做什么，再执行。
- 完成后说明改动了哪些文件、生成了哪些结果、存放在哪里。
- 如果信息不确定，先检查本地文件；无法确认时明确说明假设。

## 执行前原则

开始执行任务前，应先确认项目现状，尤其是：

- 当前是否存在 Git 仓库。
- 用户是否新增了脚本、素材、规则、模板或目录。
- `17_cherry_pipeline/` 内的代码、输入、输出是否已被更新。
- 新增文件是否会影响既有流程、模板或输出结构。

除非用户明确要求，否则不要直接重构、迁移或归档已有核心文件。

## 删除与覆盖限制

禁止批量删除文件或目录。

不要使用：

- `del /s`
- `rd /s`
- `rmdir /s`
- `Remove-Item -Recurse`
- `rm -rf`

需要删除文件时，只能一次删除一个明确路径的文件。

正确示例：

```powershell
Remove-Item "C:\path\to\file.txt"
```

如果需要批量删除文件，应停止操作，并让用户手动删除或明确授权后再处理。

覆盖文件前必须确认：

- 是否为核心文件。
- 是否会影响现有 pipeline。
- 是否已有用户手动修改。
- 是否需要先备份或生成新版本。

## 核心 Pipeline 规则

核心脚本：

```text
17_cherry_pipeline/cherry_pipeline.py
```

当前核心数据流：

```text
一集脚本 + assets_manifest.json
→ stage1_shot_block_plan_request.md
→ model_shot_block_plan.json
→ shot_block_plan.json
→ storyboard_prompt.txt
→ seedance_prompt.txt
```

关键原则：

- `shot_block_plan.json` 是唯一中间蓝图。
- 多宫格草图提示词和视频提示词必须来自同一份 block JSON。
- 不允许让草图和视频各自独立理解剧情。
- 每个 block 必须明确：
  - `duration`
  - `scene_location`
  - `characters`
  - `dialogue_lines`
  - `panel_count`
  - `panel_plan`
- `panel_plan` 是核心，必须逐格写清：
  - 可见动作
  - 机位
  - 人物位置
  - 环境动态
  - 台词归属
  - 观众必须看懂的信息

## 分镜草图提示词规则

多宫格草图必须遵守：

- 实际故事板绘图必须仅为黑白。
- 使用粗糙铅笔线条、最小细节、快速手势绘图能量、简单解剖结构、强轮廓可读性。
- 保持轻量、动态、未完成，像早期打斗编舞预可视化。
- 涉及动作、冲突、奔袭、转身、递接、攻击、躲避、追逐、法术/能量释放的 block，必须直接从动作开始。
- 不要以平静站姿、准备镜头或缓慢介绍开始。
- 宫格数必须根据动作复杂度变化，不允许整集机械使用同一种宫格数。

常用宫格：

- `4`：简单情绪或单一动作。
- `6`：常规对话、走位、递接、情绪转折。
- `8`：追逐、冲突、显现、揭露、动作递进。
- `9/12`：高潮、强动作、多层视觉信息、结尾钩子。

草图允许出现：

- 宫格
- 编号
- 镜头参数
- 彩色箭头
- 底部图例
- 构图辅助线

但这些只属于草图阶段。

## 视频提示词规则

视频提示词必须：

- 严格继承同一 block 的分镜草图。
- 逐格继承动作、机位、人物位置、环境动态和台词归属。
- 保持真人写实电影画面。
- 禁止出现“暗黑奇幻”字样。
- 视频提示词目标长度控制在 1900-2000 字之间。

如果平台限制更严格，应优先保留：

- 人物一致性
- 逐格动作
- 机位
- 人物位置
- 环境动态
- 台词归属
- 光影要求
- 禁止草图元素进入视频的约束

视频画面中严禁出现：

- 草图线稿
- 铅笔纹理
- 宫格边框
- 编号
- 镜头参数文字
- 底部图例
- 彩色箭头
- 虚线箭头
- 构图辅助线
- UI
- 水印
- 字幕
- 说明文字

分镜草图只作为动作、构图、人物位置和镜头运动参考，不得被视频模型当成画面内容。

## 光影与镜头要求

视频提示词必须强调：

- 现场拍摄感。
- 强方向主光。
- 人物脸部和手部清晰柔和补光。
- 眼神光。
- 鼻梁、颧骨、嘴唇、手背、关键道具边缘高光。
- 轮廓光将人物从背景中剥离。
- 脸、眼、嘴、手、关键道具始终优先对焦。
- 背景只提供纵深、战场氛围、烟尘、风沙、火光、压迫感。
- 避免平光、灰雾、脸部欠曝、五官糊、背景抢焦。

## 素材管理规则

每集建议独立建立素材目录，至少包含：

```text
EPXX/
  EPXX_script.txt
  assets_manifest.json
  characters/
  scenes/
  props/
```

`assets_manifest.json` 必须描述：

- 人物名
- 参考图路径
- 图像标签
- 外观描述
- 场景参考
- 道具参考

素材使用原则：

- 人物、场景、道具以 manifest 为准。
- 不得引入本集未登记元素。
- 如果图片缺失或 manifest 路径错误，应先报告并修正。
- 不应把所有素材都塞进每个 block，只写本 block 实际出现的角色、场景、道具。

## 生成图片规则

当用户要求直接生成分镜草图图片时：

- 先加载对应人物、场景、道具参考图。
- 再读取对应 `storyboard_prompt.txt`。
- 每次按 block 单独生成。
- 多张图片需要按顺序生成，不要并行混乱。
- 对暴力战场画面，避免血腥细节，用混乱、倒地、烟尘、断阵、压迫感表达。
- 生成结果优先作为预览，除非用户要求整理到项目目录。

## Git 与版本管理

执行 Git 操作前必须先确认：

- 当前目录是否是 Git 仓库。
- 是否存在远程仓库。
- 是否有未提交改动。
- 是否有用户未授权的改动。

如果当前目录不是 Git 仓库，不要假装已上传。应向用户索要远程仓库地址，或确认是否初始化新仓库。

提交前应说明：

- 将提交哪些文件。
- 是否包含输出结果、素材、缓存或生成图片。
- 是否需要排除 `__pycache__`、临时文件、大型图片或视频文件。

不要提交用户未确认的大型素材或临时生成文件。

## 修改策略

对代码、模板、规则的修改应保持小范围、可追踪。

优先顺序：

1. 读取现有文件。
2. 判断是否影响现有流程。
3. 给出简要方案。
4. 修改最小必要范围。
5. 运行验证命令。
6. 汇报结果和文件位置。

涉及 `17_cherry_pipeline/cherry_pipeline.py` 时，修改后至少运行：

```powershell
python -m compileall .\17_cherry_pipeline\cherry_pipeline.py
```

如修改视频提示词逻辑，应抽查生成结果字数和内容密度。

## 长期经验沉淀

以下经验应长期保留：

- 草图提示词和视频提示词必须来自同一份结构化分镜蓝图。
- 不能让两个智能体分别独立理解剧情。
- 动作 block 第一帧必须直接进入动作。
- 草图可以有箭头、编号和图例，但视频必须彻底禁止这些草图元素。
- 视频提示词不能压缩成摘要，必须在平台限制内尽量详细。
- 即梦平台视频提示词建议控制在 1900-2000 字。
- 光影、补光、脸手对焦是视频生成质量的关键。
- 新增目录结构不等于迁移核心 pipeline，迁移必须单独确认。
- 用户审核是流程的一部分，不应绕过。

## 协作边界

Codex 可以主动执行：

- 检查文件结构。
- 校验 manifest。
- 生成请求文件。
- 渲染本地提示词。
- 统计字数。
- 检查 pipeline 是否能编译。
- 生成分镜草图图片。
- 汇总测试结果。

Codex 不应擅自执行：

- 批量删除。
- 大范围迁移目录。
- 覆盖核心规则文件。
- 上传 Git 仓库。
- 提交大型素材。
- 修改用户未确认的长期规则。
- 将临时测试结论写入正式规则。

如需执行上述动作，必须先获得用户明确授权。
